# Frontend Redesign — Design Spec
**Date:** 2026-04-07  
**Revised:** 2026-04-07 (post three-agent review)

## Overview

Full redesign of the study app from a static vanilla JS SPA into a full-stack web application. The new app is a curriculum-based study tool organized into four tracks, where each module builds on prior ones, items are individually checkable (with link-outs for assignments), and progress is persisted per user account.

The Python data pipeline remains untouched as a standalone data generation tool. The web stack is built fresh alongside it.

---

## Stack

| Layer | Technology |
|-------|-----------|
| Frontend | React + Vite (TypeScript) |
| Backend | Hono (Node.js, TypeScript) |
| Database | SQLite via `better-sqlite3` (WAL mode, single-user/self-hosted target) |
| Auth | Email/password + Google OAuth + GitHub OAuth via `arctic` |
| Content proxy | Server-side fetch, LRU in-memory cache |

The backend serves the React build as static files in production. In dev, Vite runs on port 5173 and Hono on port 3000 with a proxy.

SQLite with WAL mode is intentional — this is a personal study tool first. The schema is straightforward to migrate to Postgres when needed.

## Tracks

Each module in `build_study_data.py` receives a `track` field. The four tracks and their module assignments:

| Track ID | Label | Module IDs |
|----------|-------|------------|
| `dsa-leetcode` | DSA & LeetCode | `setup-habits`, `big-o`, `arrays-linked-lists`, `stacks-queues-hashes`, `search-bitwise`, `trees-heaps`, `sorting`, `graphs`, `recursion-dp`, `systems-basics`, all `leetcode-export-*` modules |
| `system-design` | System Design | `system-design`, all `supplemental-systems-*` modules |
| `machine-learning` | Machine Learning | all `supplemental-ml-*` modules |
| `resume-behavioral` | Resume & Behavioral | `review-interview`, all `supplemental-interview-*` modules |

The existing `phase` field (`Start Here`, `Core Track`, etc.) is preserved on each module — it controls ordering within a track, not track membership. `PIPELINE_ONLY_MODULES` are excluded from `curriculum.json` entirely.

Any module not explicitly assigned a track defaults to `dsa-leetcode`. Track and phase assignments are the only additions to the `MODULES` schema in `build_study_data.py`.

## `curriculum.json` Schema

`build_study_data.py` is updated to write `curriculum.json` (plain JSON, no `window.` global) alongside the existing `data.js`. This is a **required code change** to `build_study_data.py`.

```jsonc
{
  "version": 1,
  "generated_at": "2026-04-07T00:00:00Z",
  "tracks": [
    {
      "id": "dsa-leetcode",
      "label": "DSA & LeetCode"
    }
    // ...
  ],
  "modules": [
    {
      "id": "arrays-linked-lists",         // stable, dash-separated
      "title": "Linear structures: arrays and linked lists",
      "track": "dsa-leetcode",
      "phase": "Core Track",               // preserved from existing field
      "summary": "...",
      "estimate": "2–3 sessions",
      "sessions": 3,
      "countsTowardSchedule": true,
      "sourceUrl": "https://...",
      "items": [
        {
          "id": "arrays-linked-lists:read:0",  // stable string ID — see Progress Keys
          "type": "read",                       // "read" | "do" | "check"
          "label": "Read: Dynamic Arrays",
          "url": "https://..."                  // null for "check" items
        }
        // ...
      ],
      "prerequisiteModuleIds": ["big-o"]    // derived from knowledge-base.json at build time
    }
    // ...
  ]
}
```

### Item IDs and Progress Keys

Each item has a **stable string ID** of the form `<moduleId>:<type>:<position>` (e.g. `arrays-linked-lists:read:0`). These IDs are stable as long as the item type and relative position within its type group don't change. They are generated deterministically by `build_study_data.py`.

The `progress` table keys on `item_id TEXT` (not `item_index INTEGER`). If module items are reordered or new items inserted, existing progress rows are unaffected — they simply refer to items by their stable ID.

### Item Type Mapping

The existing module fields map to item types as follows:

| Source field | Item `type` | Behavior |
|---|---|---|
| `resources[]` | `read` | Proxied inline if possible, link-out fallback |
| `items[]` (checklist) | `do` | Always link-out (opens new tab); LeetCode never proxied |
| `checks[]` | `check` | No link, checkbox only |

## Views

### Dashboard (`/`)
- **"What to do now" card** — top of page. Selection algorithm: find the module with the most recent `progress.updated_at` timestamp across all tracks that is not yet `done`. If no in-progress module exists, pick the first `available` module in `dsa-leetcode`. Shows the module title, track, next unchecked item, one-click to open.
- **4 track columns** — each shows track name, overall progress bar, and the next 3–4 modules with status chips (Done / In Progress / Available / Soft-locked).
- Clicking a track column navigates to `/track/:trackId`.
- **Responsive**: 4-col grid → 2-col → 1-col on smaller screens.

### Track View (`/track/:trackId`)
- Minimal top bar: app name, track name, back to dashboard.
- Full-width vertical list of modules in curriculum order (phase ordering preserved).
- Each module card: title, phase, status chip, estimated sessions, "builds on X" label if prerequisite exists.
- Soft-locked modules: dimmed, prerequisite named, still clickable.
- Clicking a module navigates to its module page.

### Module View (`/track/:trackId/module/:moduleId`)
- Minimal top bar: track name, module title, prev/next module navigation (confined to current track; last module shows "Track complete" state with prompt to switch tracks).
- **Desktop — two columns:**
  - Left: **Read** — inline markdown rendered from `study_guide_markdown` for each mapped topic. "Go deeper" external links listed below. If no topics are mapped: "Content not yet generated — run the enrichment pipeline and restart the server."
  - Right: **Do** — `do`-type items as styled link-out cards; `check`-type items as self-assessment checkboxes.
- **Mobile — tab switcher**: "Read" tab and "Do" tab toggle between the two columns. Prevents checklist from being buried below long content.
- Each item is independently checkable. Checking/unchecking calls `PUT /api/progress/:moduleId/:itemId` immediately (optimistic update).
- Notes textarea at the bottom. **Autosaves** with 1-second debounce after keyup. Shows "Saved" / "Saving…" indicator.
- Nav chrome shrinks on this page — maximum space given to content.

### Session Expiry
When any API call returns 401, the client shows a non-destructive banner: "Session expired — log back in." Clicking it redirects to `/login?next=<current-path>` so the user returns to exactly where they left off after re-authenticating.

## Prerequisite / Lock Model

Prerequisite logic is computed **at `curriculum.json` generation time** (in `build_study_data.py`), using the `planning_topic_edges` from `knowledge-base.json`. The rule:

> A module B is a prerequisite of module A if any planning topic mapped to B is listed as a prerequisite of any planning topic mapped to A in `planning_topic_edges`.

The `prerequisiteModuleIds` array on each module in `curriculum.json` is the result of this computation. It is the single authoritative source — the server does not re-derive prerequisites at runtime.

At request time, the server computes each module's status by checking the user's progress against the prerequisite list:

| Status | Condition |
|--------|-----------|
| `done` | All items complete |
| `in-progress` | At least one item complete, not all |
| `available` | No prerequisites, or all prerequisites are `done` |
| `soft-locked` | At least one prerequisite is not `done` |

Soft-locked modules are accessible — the user is never hard-blocked. No lock logic lives in the frontend.

## API

All routes under `/api`. Auth required on all routes except `/api/auth/*`.

```
POST   /api/auth/signup
POST   /api/auth/login
POST   /api/auth/logout
GET    /api/auth/me
POST   /api/auth/refresh

GET    /api/auth/oauth/google
GET    /api/auth/oauth/google/callback
GET    /api/auth/oauth/github
GET    /api/auth/oauth/github/callback

GET    /api/curriculum
  # Returns: tracks[], modules[] (with status per module, WITHOUT study_guide_markdown)

GET    /api/module/:moduleId/content
  # Returns: mapped topics with study_guide_markdown, item list
  # Called lazily when module view opens

GET    /api/progress
PUT    /api/progress/:moduleId/:itemId      # toggle item — itemId is the stable string ID

GET    /api/notes/:moduleId
PUT    /api/notes/:moduleId                 # body.content max 50,000 chars; 413 if exceeded

GET    /api/proxy?url=<encoded-url>         # server-side content fetch
```

`GET /api/curriculum` returns module list + statuses only — no markdown bodies. This keeps the payload small. `study_guide_markdown` is loaded lazily per module via `/api/module/:moduleId/content`.

## Database Schema

```sql
CREATE TABLE users (
  id            INTEGER PRIMARY KEY,
  email         TEXT UNIQUE NOT NULL,
  password_hash TEXT,            -- NULL if OAuth-only account
  created_at    TEXT DEFAULT (datetime('now'))
);

CREATE TABLE oauth_accounts (
  id                INTEGER PRIMARY KEY,
  user_id           INTEGER NOT NULL REFERENCES users(id),
  provider          TEXT NOT NULL,   -- 'google' | 'github'
  provider_user_id  TEXT NOT NULL,
  created_at        TEXT DEFAULT (datetime('now')),
  UNIQUE(provider, provider_user_id)
);

CREATE TABLE sessions (
  id            TEXT PRIMARY KEY,   -- opaque refresh token ID (UUID)
  user_id       INTEGER NOT NULL REFERENCES users(id),
  created_at    TEXT DEFAULT (datetime('now')),
  expires_at    TEXT NOT NULL,
  revoked       INTEGER DEFAULT 0
);

CREATE TABLE progress (
  id           INTEGER PRIMARY KEY,
  user_id      INTEGER NOT NULL REFERENCES users(id),
  module_id    TEXT NOT NULL,
  item_id      TEXT NOT NULL,       -- stable string ID, e.g. "arrays-linked-lists:read:0"
  item_type    TEXT NOT NULL,       -- 'read' | 'do' | 'check'
  completed    INTEGER DEFAULT 0,
  updated_at   TEXT DEFAULT (datetime('now')),
  UNIQUE(user_id, module_id, item_id)
);

CREATE TABLE notes (
  id         INTEGER PRIMARY KEY,
  user_id    INTEGER NOT NULL REFERENCES users(id),
  module_id  TEXT NOT NULL,
  content    TEXT DEFAULT '',
  updated_at TEXT DEFAULT (datetime('now')),
  UNIQUE(user_id, module_id)
);
```

## Auth Logic

### Token Model
- **Access token**: short-lived JWT (15 minutes), stored in httpOnly cookie (`SameSite=Lax`). `SameSite=Lax` (not `Strict`) is required for OAuth callback redirects to work.
- **Refresh token**: opaque UUID stored in the `sessions` table and in a second httpOnly cookie. Valid for 30 days. `POST /api/auth/refresh` issues a new access token and rotates the refresh token (old session row revoked, new one inserted).
- **Logout**: revokes the session row. Stolen tokens are invalidated immediately.
- **Password change**: revokes all sessions for the user.

### OAuth Merge Rules
- **Sign up with Google/GitHub**: look up existing account by email.
  - If email is **unverified** by the provider: reject with error "Could not verify your email — try signing in with a password instead." Never auto-merge on an unverified email.
  - If no existing account: create account with `password_hash = NULL`, insert `oauth_accounts` row.
  - If existing account found (verified email match): link OAuth to existing account (insert `oauth_accounts` row). This is the auto-merge path — only taken on a verified provider email.
- **Sign in with email, no password set**: return error code `NO_PASSWORD`. Client shows: "Looks like you usually sign in with Google or GitHub. Want to set a password?" — sends a password-set link to the user's email.
- **Sign in with email, password set**: standard bcrypt comparison.

## Content Proxy

`GET /api/proxy?url=<encoded-url>`

### SSRF Prevention
The server validates the requested URL against a **domain allowlist** before issuing any fetch. The allowlist is defined in server config and populated with the domains present in `knowledge-base.json` source URLs (e.g. `wikipedia.org`, `geeksforgeeks.org`, `freecodecamp.org`, `github.com`, `startupnextdoor.com`, etc.). Requests to any domain not on the list return 403. Requests to RFC-1918 ranges, loopback, or link-local addresses are rejected regardless of the allowlist.

LeetCode domains (`leetcode.com`) are explicitly excluded from the allowlist — LeetCode items always link out.

### Extraction & Format
Content is extracted using `@mozilla/readability` (Node.js port). The extracted article is converted to Markdown via `turndown`. The proxy endpoint returns `{ markdown: string }`. This matches the format of `study_guide_markdown` in `knowledge-base.json`, so the client renders both with the same component.

### Cache
LRU cache (`lru-cache` package), max 150 entries, TTL 1 hour. Cache keys are the normalized URL. On cache miss or fetch failure, the response is `{ error: "unavailable" }` — the client falls back to a plain link-out card.

## Curriculum Loading

At server startup:
1. Read `curriculum.json` — validate `version` field matches `CURRICULUM_JSON_VERSION` constant in server. Fail fast with a descriptive error on mismatch.
2. Read `knowledge-base.json` — validate `version` field matches `KNOWLEDGE_BASE_VERSION` constant. Fail fast on mismatch.
3. Build in-memory index: module ID → module, topic ID → topic, module ID → mapped topics.
4. This object is read-only at runtime.

To update: run `python3 build_study_data.py` and/or `python3 pipeline/runner.py` → restart server.

## What the Python Pipeline Stays Responsible For

- Scraping, embedding, clustering, synthesizing resource content, writing `knowledge-base.json`.
- `build_study_data.py` gains one new responsibility: writing `curriculum.json`. No other pipeline changes.
- Run manually; the web server has no dependency on running the pipeline at runtime.
